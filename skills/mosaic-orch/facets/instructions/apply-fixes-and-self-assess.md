# 指示: レビュー修正 + 自己採点

前回のレビュー指摘を修正し、同じ5軸rubricで自己採点してください。

## デバッグ手順（修正前に必ず実行）

レビュー指摘を闇雲に直す前に、以下のステップで原因を特定する:

### 1. 指摘を分類する
- **明確な修正**: 「N+1がある」「命名がXでなくYであるべき」→ 直接修正
- **設計の指摘**: 「責務が混在している」「依存方向が逆」→ 原因分析が必要
- **曖昧な指摘**: 「もう少し良くできそう」→ 具体化を求める（修正しない）

### 2. 原因を特定する（設計の指摘の場合）
1. 指摘箇所の周辺コードを読む（前後50行）
2. なぜこの設計になったか推測する
3. 指摘の根本原因を1文で言語化する
4. 修正方法を2つ以上考える
5. 最もシンプルな方法を選ぶ

### 3. 修正の影響範囲を確認する
- 修正するファイルを列挙する
- 修正が他の機能に影響しないか確認する
- 影響がある場合、その範囲もテストする

## 手順

### Step 1: 指摘内容の確認

入力として渡されるレビュー結果から、修正必須項目（Issues）を確認する。

### Step 2: 修正実装

各指摘に対して修正を実施する:
- **ファイル名:行番号** の指摘に対して、指示された方向性で修正
- 修正が別の問題を引き起こさないか確認する
- 修正後、関連テストを再実行する

### Step 3: ビルド・テスト確認

```bash
# プロジェクトの標準ビルド・テストコマンドを実行
./gradlew compileKotlin && ./gradlew test  # BE
npm run build && npm run test              # FE
```

### Step 4: 自己採点

Rubrics セクションに提供される5軸に従い、修正後のコードを自己採点する:
- 各軸の評価（C/B/A/A+/S）
- 評価の根拠（修正前後の比較を含む）
- 総合グレードの算出

### Step 5: ローカルコミット

```bash
git add {修正ファイル}
git commit -m "fix: address review feedback — {修正概要}"
```
