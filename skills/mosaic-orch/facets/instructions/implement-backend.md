# 指示: バックエンド実装

Kotlinでのバックエンド実装を行ってください。

## 実装前の設計確認（5分以内）

コードを書く前に以下を確認する:

1. **既存パターンの確認**: 同種の機能がどう実装されているか、既存コードを2-3箇所読む
2. **変更ファイルの特定**: どのファイルを変更するか列挙する（新規作成するファイルも）
3. **依存方向の確認**: 変更がレイヤー境界を跨ぐ場合、依存方向が正しいか確認する
4. **テスト方針の決定**: 何をテストするか、どのレベル（ユニット/統合）でテストするか決める

## 手順

### Step 1: 既存コード調査

- Input に Codebase Context が含まれている場合、それを基に調査範囲を絞る
- 関連する既存コードのパターン（命名規約、ディレクトリ構造、エラーハンドリング方式）を調査する
- 同種の既存実装があれば、それに倣う

### Step 2: テスト先行（TDD）

- BE振る舞い変更の場合、まずテストを書く
- テスト名は振る舞いを説明する形式（「何をしたら何が起きる」）
- AAA / GWT パターンで構造化する
- **ユニットテストとHTTP統合テストの両方を書く**:
  - ユニット: ロジック層の関数を直接テスト
  - 統合: HTTPリクエスト→レスポンスの全経路テスト（クエリパラメータ、ステータスコード、レスポンスボディ）
  - 統合テストがないとルーティング層のバグが検出できない

### Step 3: 実装

- 既存パターンに従ってコードを書く
- 以下を意識する:
  - N+1回避（batchInsert/Upsert活用、JOINの適切な使用）
  - トランザクション境界の設計
  - バリデーションの網羅（境界値、null安全性）
  - sealed class / data class / 拡張関数の適切な活用

### Step 4: ビルド・lint・テスト実行（必須 — スキップ厳禁）

以下のコマンドを**すべて実行**し、出力をそのまま貼り付けること:

```bash
./gradlew compileKotlin
./gradlew detekt
./gradlew ktlintCheck
./gradlew test --tests "{関連テストクラス}"
```

**禁止事項:**
- ビルドが失敗したままコミットすること
- テスト0件でコミットすること
- テスト実行せずに「テストを書いた」と報告すること
- **追加した機能と無関係なテストだけ**を根拠に「テスト済み」と報告すること

**HTTP統合テスト（APIエンドポイント変更時は必須）:**
- ユニットテストだけでは不可。HTTPリクエスト→レスポンスの全経路テストを含めること
- 正常系 + 異常系（404, 400）の最低2パターン
