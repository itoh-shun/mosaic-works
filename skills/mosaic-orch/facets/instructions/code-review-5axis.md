# 指示: 5軸コードレビュー

`git diff {base-branch}...HEAD` の変更全体に対して、5軸コードレビューを実行してください。

## 手順

### Step 1: 変更差分の取得

```bash
git diff {base-branch}...HEAD
```

変更全体を把握する。ファイル数が多い場合はファイルごとに確認する。

### Step 2: 5軸レビュー実行

Rubrics セクションに提供される5つの評価軸に従って、各軸を評価する:

1. **Performance（パフォーマンス）** — rubric: performance
2. **Naming（命名・一貫性）** — rubric: naming
3. **Testing（テスト）** — rubric: testing
4. **Security（セキュリティ）** — rubric: security
5. **Design（設計・構造）** — rubric: design

各軸について:
- 採点段階に従って評価する（C/B/A/A+/S）
- 指摘がある場合は**ファイル名:行番号**で明示する
- 根拠を具体的に記述する

### Step 3: 総合評価の算出

5軸の評価から総合グレードを算出する:
- **総合グレード = 全軸の最低値**（最弱リンク方式）
- 1軸でもCなら総合C。1軸でもBなら総合B
- この方式により、1軸だけ弱い場合でもfix stageが発動する

### Step 4: 修正指示（A+未満の場合）

A+未満の軸がある場合、修正必須項目を明確に指示する:
- **ファイル名:行番号**: 指摘内容
- 修正の方向性を具体的に示す

## 各軸の詳細チェックリスト

### Performance チェック
- [ ] N+1クエリがないか（ループ内でDBアクセスしていないか）
- [ ] バッチ処理が使われるべき箇所で使われているか（batchInsert, batchUpsert）
- [ ] 不要な全件取得がないか（LIMITやページネーション）
- [ ] インデックスが効くクエリになっているか
- [ ] メモリに大量データを載せていないか

### Naming チェック
- [ ] 変数・関数名が振る舞いを正確に表現しているか
- [ ] 既存コードの命名パターンと一貫しているか
- [ ] 略語を避け、意図が伝わる名前か
- [ ] ファイル名がディレクトリ構造の規約に沿っているか
- [ ] Boolean は is/has/can/should で始まっているか

### Testing チェック
- [ ] 振る舞い変更にテストが伴っているか
- [ ] 正常系だけでなく異常系・境界値があるか
- [ ] テスト名が何をテストしているか説明しているか
- [ ] テストが実際に実行され、結果が報告されているか
- [ ] モックが適切か（外部依存のみモック、内部ロジックはモックしない）

### Security チェック
- [ ] ユーザー入力のバリデーションがあるか
- [ ] 認可チェックが適切か（他ユーザーのデータにアクセスできないか）
- [ ] SQLインジェクション、XSSの可能性がないか
- [ ] シークレット・認証情報がハードコードされていないか
- [ ] エラーメッセージに内部情報が含まれていないか

### Design チェック
- [ ] 関心の分離が守られているか（1クラス/関数が1つの責務）
- [ ] 依存方向が正しいか（ドメインが外部に依存していないか）
- [ ] 過剰抽象化がないか（YAGNI）
- [ ] 変更箇所が最小限か（必要十分な変更か）
- [ ] 既存パターンに従っているか
